# 비디오

정지 이미지를 빠르게 연속으로 보여줘서 움직임으로 인식되도록 만든 매체. 모든 비디오 기술(코덱, 컨테이너, 스트리밍)은 이 단순한 정의 위에 쌓아 올린 압축·전송 메커니즘이다.

## 1. 비디오 = 이미지의 연속

### 플립북 비유

종이책 모서리에 그림을 조금씩 다르게 그려 넣고 빠르게 넘기면 움직이는 것처럼 보인다. 비디오도 똑같다 — 정지 이미지(**프레임, frame**)를 1초에 수십 장씩 보여주면 사람의 눈은 그 사이를 매워 연속 동작으로 인식한다.

```
비디오 1초 = [프레임 1][프레임 2][프레임 3] ... [프레임 30]
              ↑           ↑           ↑
           정지 이미지   정지 이미지   정지 이미지
```

이 현상의 이름은 **잔상 효과(persistence of vision)**. 눈과 뇌가 한 이미지를 약 1/16초 동안 기억하기 때문에, 그 안에 다음 이미지가 들어오면 끊김 없이 이어진 것으로 본다.

### 프레임 레이트 (fps)

1초에 몇 장의 프레임을 보여주는가. 단위는 **fps (frames per second)**.

| fps | 어디 쓰나 |
|------|----------|
| **24** | 영화. 가장 영화적인 모션 블러 |
| **25 / 30** | TV 방송 (PAL=25, NTSC=30). 일반 콘텐츠 |
| **60** | 게임, 스포츠 중계, 부드러운 모션 |
| **120 / 240** | 슬로우 모션 촬영 후 24fps로 재생 (4~10배 슬로우) |

24fps도 충분히 자연스럽게 보이는 이유는 영화가 의도적으로 **모션 블러**를 남기기 때문. 게임은 매 프레임이 칼같이 선명해서 24fps면 끊겨 보인다 — 그래서 60fps 이상이 표준.

## 2. 한 프레임 = 픽셀의 격자

### 해상도(resolution)

한 프레임은 작은 색점(**픽셀, pixel**)이 가로 × 세로로 빽빽이 박힌 격자다. 이 격자의 크기가 **해상도**.

```
1080p (Full HD): 1920 × 1080 = 약 207만 픽셀
4K UHD       : 3840 × 2160 = 약 829만 픽셀  (1080p의 4배)
8K UHD       : 7680 × 4320 = 약 3,318만 픽셀 (1080p의 16배)
```

**"p" vs "i"**:
- `1080p` — progressive scan, 한 번에 1080줄 모두 그림
- `1080i` — interlaced scan, 홀수줄·짝수줄을 번갈아 그림 (구식 TV 잔재)

현대 디지털 비디오는 거의 다 progressive. interlace는 방송 표준 호환성 때문에 일부 남아 있다.

### 종횡비(aspect ratio)

가로:세로 비율.

| 비율 | 어디 |
|------|------|
| **16:9** | 현대 TV·모니터 표준, 1080p/4K가 모두 16:9 |
| **9:16** | 세로 영상 (TikTok, Shorts, Reels) |
| **2.39:1** | 시네마스코프 영화 |
| **4:3** | 구식 TV·모니터, 거의 사라짐 |

## 3. 픽셀 = 색을 어떻게 표현하나

### 색 공간(color space) — RGB vs YUV

**RGB**: 빨강·초록·파랑 세 채널. 모니터·카메라 센서가 직접 다루는 자연스러운 표현.

**YUV** (또는 YCbCr): 밝기(Y) + 색차 두 채널(U, V). 비디오 코덱이 거의 항상 쓰는 표현.

왜 YUV를 쓰나? 사람의 눈은 **밝기 차이에 민감하고 색 차이엔 둔감**하다 (간상세포가 원추세포보다 약 20배 많다). 그래서 색 채널만 절반으로 줄여도 (subsampling) 화질 저하를 거의 못 느끼면서 데이터 양은 크게 줄일 수 있다.

```
4:4:4  → Y, U, V 모두 풀 해상도 (방송 마스터, 그래픽 작업)
4:2:2  → 색 채널 가로 절반 (방송 표준)
4:2:0  → 색 채널 가로·세로 절반 (인터넷·소비자 비디오 대다수)
```

YouTube·Netflix·디스크의 동영상은 거의 다 4:2:0이다. 이미 데이터 절반을 던지고 시작한다는 뜻. 자세한 내용은 YUV 페이지에서 다룬다 (Y/U/V 의미, 변환 행렬, BT.601/709/2020 변형, 픽셀 포맷).

### 비트뎁스(bit depth)

픽셀당 색을 몇 비트로 기록하나.

| 비트뎁스 | 채널당 색 단계 | 어디 |
|---------|---------------|------|
| **8-bit** | 256단계 (2⁸) | 일반 SDR 콘텐츠 |
| **10-bit** | 1024단계 (2¹⁰) | HDR, 고품질 방송, ProRes |
| **12-bit** | 4096단계 (2¹²) | 영화 마스터, ACES |

8-bit에서 색을 부드럽게 그라데이션할 때 단계가 보이는 현상이 **밴딩(banding)**. HDR 시대에 10-bit가 표준이 되는 이유.

### HDR vs SDR

- **SDR (Standard Dynamic Range)**: 8-bit + Rec.709 색역. 오랜 표준
- **HDR (High Dynamic Range)**: 10-bit+ + Rec.2020 색역 + PQ/HLG 전송 함수. 더 넓은 밝기·색

HDR은 비트뎁스만 늘리는 게 아니라 **메타데이터로 밝기 매핑 정보**를 같이 보낸다 (Dolby Vision, HDR10, HDR10+). 이 메타데이터가 FFmpeg 8.1의 LCEVC 메타데이터 처리, EXIF 파싱 같은 기능과 연결된다.

## 4. raw 데이터 양 — 왜 코덱이 필수인가

이제 위 모든 걸 곱해 raw 데이터 양을 계산해보면 압축이 왜 필수인지 한눈에 보인다.

### 1080p 60fps 4:2:0 8-bit 1초 분량

```
픽셀 수    : 1920 × 1080 = 2,073,600
4:2:0 평균 : 픽셀당 1.5 바이트 (Y 1바이트 + U·V 합쳐서 0.5바이트)
한 프레임  : 2,073,600 × 1.5 = 약 3.1 MB
1초 (60fps): 3.1 MB × 60 = 약 186 MB
1분        : 186 × 60 = 약 11 GB
1시간 영화 : 약 670 GB
```

압축 안 한 1시간짜리 1080p 60fps 영화는 **670GB**. 4K로 가면 4배, 8K로 가면 16배. 인터넷으로 전송은커녕 디스크에 담기도 힘들다.

### 코덱이 줄이는 양

| 코덱 | 1080p 60fps 1초 |
|------|-----------------|
| raw | 186 MB |
| H.264 (5 Mbps) | 0.625 MB → **300배 압축** |
| H.265 (3 Mbps) | 0.375 MB → 500배 압축 |
| AV1 (2.5 Mbps) | 0.3 MB → 600배 압축 |
| VVC (1.5 Mbps) | 0.2 MB → 900배 압축 |

코덱은 압축률을 점점 올리는 게 한 세대씩의 목표다. H.264 → H.265는 약 50% 추가 압축, H.265 → VVC도 약 50%.

## 5. 코덱은 어떻게 그렇게 줄이나

비디오는 **시간적·공간적 중복(redundancy)**이 매우 크다. 코덱은 이 중복을 제거하는 기술이다.

### 공간적 중복 (Intra prediction)

한 프레임 안에서 옆 픽셀끼리는 비슷하다. 푸른 하늘 영역은 거의 같은 색이 수천 픽셀 이어진다.

→ "이 블록은 위쪽 블록과 거의 같다, 차이만 기록"

### 시간적 중복 (Inter prediction)

연속한 프레임끼리는 거의 같다. 1/30초 사이엔 큰 변화가 없다.

→ "이 프레임은 앞 프레임에서 사람만 5픽셀 오른쪽으로 움직였다, 그 차이만 기록"

이를 **모션 추정(motion estimation)** 이라 한다. 코덱 인코딩에서 가장 비싼 단계 — 어디가 움직였는지 찾는 게 무거워서 인코딩 시간 대부분을 차지한다.

### 키프레임(keyframe, I-frame)

매 프레임이 앞 프레임에 의존하면 중간부터 재생할 수 없다. 그래서 코덱은 **주기적으로 완전한 프레임(I-frame, keyframe)** 을 끼워 넣는다. 보통 2~10초마다.

이 키프레임 위치가 HLS 세그먼트 분할의 기준점이 된다 — 세그먼트는 항상 키프레임에서 시작해야 독립 재생이 가능하다.

### 직렬 vs 병렬의 충돌

이 압축 방식들이 GPU 가속을 어렵게 만드는 원인이다:
- **직렬 의존**: 한 블록을 풀려면 옆 블록을 먼저 풀어야 함 (intra prediction)
- **병렬 가능**: 양자화, 변환 등은 블록별 독립

GPU는 병렬에 강하고 직렬에 약하다. 이 충돌을 어떻게 푸느냐가 Vulkan Compute 코덱이 다루는 문제 (FFmpeg Vulkan 시대 참고).

## 6. 비디오에서 스트리밍으로

비디오 자체의 개념(이 페이지)은 한 파일·한 디스크의 이야기다. 이를 네트워크 너머의 시청자에게 전달하는 순간 **스트리밍** 영역으로 넘어간다 — 컨테이너 포맷, 세그먼트 분할, 프로토콜, ABR이 모두 거기서부터 등장한다.

다음으로 읽을 것: 비디오 스트리밍 → 비디오-스트리밍-입문-가이드

## 관련 개념
- YUV — 비디오 코덱이 RGB가 아닌 YUV를 쓰는 이유와 subsampling 상세
- 비디오 스트리밍 — 비디오를 네트워크로 전달하는 기술
- 비디오-스트리밍-입문-가이드 — 코덱·컨테이너·프로토콜까지 단계별 가이드
- FFmpeg — 비디오의 인코딩·디코딩·변환 표준 도구
- Vulkan — 코덱의 GPU 가속을 지원하는 컴퓨트 API
