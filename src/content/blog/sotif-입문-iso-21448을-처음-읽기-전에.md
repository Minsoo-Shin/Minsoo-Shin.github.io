# SOTIF 입문 가이드

ISO 21448(SOTIF)을 처음 읽기 전에, 쉬운 개념부터 쌓아올리는 가이드.

## 1단계: 왜 SOTIF가 필요한가

### 자동차 안전의 두 가지 질문

```
질문 A: "브레이크가 고장나면 어떻게 되지?"    → ISO 26262 (기능 안전)
질문 B: "브레이크는 멀쩡한데 잘못 밟으면?"    → ISO 21448 (SOTIF)
```

ISO 26262는 **고장**을 다룬다. 회로가 단선되거나, 소프트웨어 버그가 있거나. 수십 년간 자동차 안전은 이 관점이었다.

그런데 자율주행이 등장하면서 새로운 문제가 생겼다: **시스템이 완벽히 정상 동작하는데도 위험한 상황**. 카메라가 정상이고 알고리즘도 설계대로 돌아가는데, 밝은 하늘에 흰색 트럭을 못 보는 것. 이건 "고장"이 아니라 **능력의 한계**다.

SOTIF는 이 한계를 다룬다.

## 2단계: 핵심 개념 4개

SOTIF를 이해하려면 딱 네 가지 개념만 먼저 잡으면 된다.

### (1) Functional Insufficiency — 기능 불충분

시스템이 설계대로 동작하지만 **부족한 부분**이 있다는 뜻.

```
일상 비유: 우산은 정상이지만, 강풍에는 뒤집어진다.
           우산의 "고장"이 아니라 우산의 "한계".

자동차 예시:
- 카메라가 역광에서 물체를 못 봄 (센서 성능 한계)
- 알고리즘이 스케이트보드 탄 사람을 "보행자"로 인식하되 
  속도가 비정상이라 무시 (알고리즘 논리 한계)
- 사양에 "공사구간 대응"이 빠져 있음 (사양 누락)
```

두 종류가 있다:
- **Insufficiency of specification** — 사양 자체가 빠졌거나 부족
- **Performance insufficiency** — 사양은 있지만 구현이 성능 목표에 못 미침

### (2) Triggering Condition — 트리거 조건

기능 불충분을 **실제로 활성화**시키는 환경 조건.

```
일상 비유: 우산의 한계(강풍에 뒤집힘)를 활성화하는 "강풍" 자체가 트리거 조건.
           강풍이 없으면 우산의 한계는 문제가 되지 않는다.

자동차 예시:
- "역광"이 카메라 인식 한계를 활성화
- "폭우 + 야간"이 라이더 성능 저하를 활성화
- "도로에 그려진 3D 착시 횡단보도"가 거짓 양성 제동을 활성화
```

핵심 통찰: **기능 불충분 × 트리거 조건 = 위험 행동**. 둘 중 하나만 있으면 안전하다. 둘이 만나야 위험해진다.

### (3) Hazardous Behaviour — 위험 행동

기능 불충분이 트리거 조건에 의해 활성화되어 **차량 수준에서 나타나는 위험한 동작**.

```
자동차 예시:
- 앞차를 못 보고 제동하지 않음 (false negative)
- 없는 장애물에 급제동 (false positive → 후방 추돌 위험)
- 차선을 잘못 인식해서 옆 차선으로 이탈
```

위험 행동 자체가 반드시 사고는 아니다. 후속 차량이 없으면 급제동해도 괜찮다. 위험 행동 + 특정 시나리오(후속 차량 근접) = 사고.

### (4) Reasonably Foreseeable Misuse — 합리적으로 예측 가능한 오용

사람이 시스템을 잘못 사용하는 것. 표준은 이것도 **시스템이 대비해야 할 범위**로 본다.

```
일상 비유: 전자레인지에 금속 그릇 넣지 말라고 써있지만, 
           사람들이 넣을 수 있다는 걸 예측하고 감지·차단하는 것.
           (의도적 분해는 범위 밖)

두 종류:
- 직접 오용: 고속도로 전용 기능을 도심에서 켬 → 위험을 직접 유발
- 간접 오용: 시스템이 위험 행동 중인데 운전자가 졸고 있음 → 제어 실패
```

## 3단계: 4분면 모델

네 개념을 알았으면, 이제 SOTIF의 큰 그림을 볼 수 있다.

```
              Known              Unknown
         ┌─────────────────┬─────────────────┐
         │                 │                 │
   Not   │    Area 1       │    Area 4       │
Hazardous│  (안전 확인됨)    │  (몰랐지만 안전) │
         │    ✓ 최대화      │                 │
         ├─────────────────┼─────────────────┤
         │                 │                 │
Hazardous│    Area 2       │    Area 3       │
         │  (위험 식별됨)    │  (미발견 위험)   │
         │    → 설계로 해결  │    → 테스트로 축소│
         └─────────────────┴─────────────────┘
```

- **Area 2 줄이기**: 위험을 알고 있으니 설계를 고치거나 기능을 제한
- **Area 3 줄이기**: 아직 모르는 위험이니 다양한 테스트로 발견해서 Area 2로 옮기기
- 최종 목표: Area 2+3의 **잔여 위험**이 수용 가능 수준임을 논증

## 4단계: 프로세스 흐름

4분면을 줄이기 위한 표준의 작업 순서. 반복(iteration)이 핵심이다.

### 자동차 공급망 구조

프로세스를 이해하려면 먼저 자동차 산업의 계층 구조를 알아야 한다:

```
OEM (완성차 제조사: 현대, BMW, 테슬라)
 │ → 차량 전체를 설계하고 최종 조립·판매
 ├── Tier 1 (1차 협력사: 모비스, 보쉬, 콘티넨탈)
 │    → OEM에 직접 납품 (자율주행 모듈, 카메라, 레이더)
 └── Tier N (2차+ 협력사: ON Semi, Infineon)
      → Tier 1에 부품/칩/SW 공급 (이미지 센서 칩, GPU)
```

SOTIF 문제는 한 회사 안에서 완결되지 않는다. Tier N의 센서 성능 한계가 → Tier 1의 알고리즘 오판을 일으키고 → OEM의 차량 수준에서 위험 행동으로 나타난다. 그래서 표준은 **각 계층의 성능 한계와 오용 가능성이 하위에서 상위로 매 반복마다 전달**되어야 한다고 요구한다.

### Clause 6: S=0/C=0 판단이란

설계 루프의 Clause 6에서 나오는 "S=0 또는 C=0이면 OK"가 뜻하는 것:

- **S(Severity)** = 사고 시 피해 심각도. S=0이면 "이 위험 행동으로는 아무도 안 다침"
- **C(Controllability)** = 사람이 통제 가능한 정도. C=0이면 "위험하더라도 쉽게 통제 가능"

```
위험 행동 식별 → S=0? → Yes → 종료 (안전)
                  │No
                  → C=0? → Yes → 종료 (안전)
                    │No
                    → SOTIF 해저드로 등록 → Clause 7 진행
```

**예시로 이해하기:**
- 저속 5km/h 주차 보조가 10cm 벗어남 → S=0 (다칠 일 없음) → **끝**
- L2 차선 유지가 서서히 옆으로 밀림 → S>0 (옆 차 충돌 가능) → C=0 (운전자가 충분히 교정 가능) → **끝**
- AEB가 고속에서 불필요한 급제동 → S>0 (후방 추돌) → C>0 (뒤차가 못 피함) → **SOTIF 해저드** → 계속 분석

### Clause 7: "왜 위험한가?" 원인 추적

Clause 6에서 SOTIF 해저드로 등록된 위험에 대해, **정확히 어떤 약점이 어떤 조건에서 활성화되는지** 파헤친다.

```
Clause 6: "AEB 급제동이 위험하다" (무엇이 위험한가)
    ↓
Clause 7: "왜 급제동이 발생하는가?" (원인 추적)
    ↓
  → 카메라가 그림자를 객체로 오인 (functional insufficiency)
  → 오후 역광 조건에서 활성화 (triggering condition)
  → 조합: 역광 + 젖은 노면 반사가 겹치면 더 심해짐
```

**두 방향에서 찾는다:**
- **시스템 → 시나리오**: "카메라 분류기가 작은 객체에 약하다" → "그럼 원거리 오토바이, 어린이가 위험"
- **시나리오 → 시스템**: "도로에 3D 착시 횡단보도가 있다" → "카메라는 속지만 레이더는 안 속음 → 융합으로 해결?"

**Sense·Plan·Act 각각 분석:**
- **Sense**(인지): 기상, 조도, 센서 오염, 노화 → 오분류, 거짓 양성/음성
- **Plan**(판단): 공사구간, 타 차량 비예측 행동, ML 한계 → 경로 오판, 사양 누락
- **Act**(실행): 극저온 제동 지연, 중량 변화 → 제동력 부족

핵심: **단일 트리거가 아니라 조합**. "폭우"만으로는 안전해도, "폭우 + 야간 + 공사구간"이 겹치면 위험.

트리거 조건에서 시스템 반응이 수용 가능하면 → Area 1(안전). 불가능하면 → Clause 8(설계 수정).

### 전체 흐름

```
┌─────────────────────────────────────────────────────┐
│                    설계 루프                          │
│                                                     │
│  Clause 5: 사양 정의 (기능, ODD, 센서, HMI)           │
│      ↓                                              │
│  Clause 6: 해저드 식별 + 위험 평가                     │
│      ↓    "S=0 또는 C=0이면 OK, 아니면 계속"           │
│  Clause 7: 기능 불충분 + 트리거 조건 분석               │
│      ↓                                              │
│  Clause 8: 설계 수정 (센서 개선, 기능 제한, HMI 개선)   │
│      ↓    Clause 5로 되돌아가서 사양 업데이트            │
└──────────────┬──────────────────────────────────────┘
               ↓
┌─────────────────────────────────────────────────────┐
│                    V&V 루프                           │
│                                                     │
│  Clause 9:  V&V 전략 정의 (테스트 방법, 목표값)         │
│      ↓                                              │
│  Clause 10: Known 시나리오 검증 (Area 2)               │
│      ↓                                              │
│  Clause 11: Unknown 시나리오 탐색 (Area 3)             │
│      ↓    새 위험 발견 시 → 설계 루프로 복귀             │
└──────────────┬──────────────────────────────────────┘
               ↓
  Clause 12: 릴리스 판단 (승인 / 조건부 승인 / 거부)
               ↓
  Clause 13: 운영 중 모니터링 (필드에서 새 위험 발견 시 → 설계 루프)
```

## 5단계: 이제 표준을 읽을 준비

개념을 잡았으면, SOTIF 위키 페이지를 이 순서로 읽는 것을 추천한다:

| 순서  | 섹션                     | 왜                      |
| --- | ---------------------- | ---------------------- |
| 1   | 핵심 정의 + ISO 26262와의 차이 | SOTIF가 다루는 영역의 경계 확인   |
| 2   | 핵심 용어 표                | 표준 전체에서 반복되는 단어들       |
| 3   | Sense-Plan-Act 모델      | 시스템을 분석하는 프레임워크        |
| 4   | 표준 구조 (13개 Clause)     | 각 절이 4분면의 어떤 부분을 담당하는지 |
| 5   | 잔여 위험 수용 원칙            | "충분히 안전하다"의 기준이 뭔지     |
| 6   | ML 고려사항                | 현대 자율주행에서 가장 뜨거운 부분    |

원본 표준(ISO 21448:2022)은 본문 Clause 4(Overview)부터 읽되, Annex B(시나리오 분석 예시)와 C.5(시뮬레이션 샘플링)가 실무적으로 가장 유용하다.
