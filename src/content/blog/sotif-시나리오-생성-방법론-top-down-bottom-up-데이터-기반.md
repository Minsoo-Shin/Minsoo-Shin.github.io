# SOTIF 시나리오 생성 방법론

SOTIF 검증의 출발점은 시나리오다. 이 문서는 시나리오의 구성요소를 분리하고, 시나리오를 체계적으로 만들어내는 세 가지 접근법과 그 결합 방식을 정리한다.

## 시나리오 구성요소

시나리오는 "상황 설정"이고, 시스템 거동은 그 상황에 넣었을 때 "관측되는 결과"다. 기능 부족은 이 둘을 잇는 인과 메커니즘이다.

```
트리거 조건                 기능 부족                     시스템 거동
(맨홀 뚜껑 반사)  →  (레이더가 장애물로 오분류)  →  (불필요한 급제동)
     원인                  내부 메커니즘                     결과
```

| 구분         | 역할                | 예시                        |
| ---------- | ----------------- | ------------------------- |
| **운행 조건**  | 배경 설정 (ODD 파라미터)  | 우천, 야간, 60km/h, 편도 2차로    |
| **트리거 조건** | 문제를 유발하는 자극       | 금속 맨홀 뚜껑, 역광, 공사 임시 차선    |
| **기능 부족**  | 시스템 내부 한계 (원인 설명) | 레이더 오탐지, 카메라 포화, 차선 인식 혼란 |
| **시스템 거동** | 관측되는 반응 (검증 대상)   | 급제동, 차선 이탈, 무반응           |

시나리오를 **만드는** 단계에서는 운행 조건 + 트리거 조건을 조합한다. 시스템 거동은 그 시나리오를 시뮬레이션이나 실차에 넣어서 **관측**하는 것이다. 기능 부족은 "이 트리거가 왜 위험할 수 있는지"에 대한 가설 역할을 한다.

## 세 가지 생성 접근법

### Top-Down (연역적): 위험에서 출발

**"어떤 사고가 날 수 있는가?"에서 역추적으로 조건을 찾는다.**

```
Vehicle-level Hazard (예: 의도치 않은 급제동)
  → 어떤 기능 오작동이 이걸 유발? (예: 오탐지 — false positive)
    → 어떤 트리거가 오탐지를 유발? (예: 금속 맨홀 뚜껑의 레이더 반사)
      → 어떤 운행 조건에서? (예: 우천 + 저속 + 시내도로)
        = 시나리오 1개 완성
```

대표 기법:
- **STPA (Systems-Theoretic Process Analysis)**: 제어 구조에서 Unsafe Control Action을 도출하고, 그 원인이 되는 시나리오를 역추적
- **FTA (Fault Tree Analysis)**: 최상위 사고 사건에서 트리 형태로 원인 조합을 분해

**역추적의 목적은 탐색이 아니라 필터링이다.** Bottom-Up으로 파라미터를 조합하면 수십만 개 시나리오가 나오지만 대부분은 무해하다. Top-Down은 "이 사고를 유발할 수 있는 조합"만 좁혀내는 근거를 만든다. 즉 인과 체인(트리거 → 기능 부족 → 위험 행동)이 있어야 어디를 집중 검증할지 판단할 수 있다.

### Bottom-Up (귀납적): 조건에서 출발

**"이 환경/입력 조건에서 시스템이 어떻게 반응하는가?"를 조합적으로 탐색한다.**

```
파라미터 공간 정의:
  - 환경: 날씨(맑음/비/눈/안개), 조도(주간/야간/역광), 노면(건조/젖음/결빙)
  - 도로: 직선/곡선, 차선 수, 구배, 교차로 유무
  - 교통 참여자: 차량/보행자/자전거, 속도, 궤적, 밀도
  - 자차 상태: 속도, 가감속, 기능 모드
  → 조합 → 시나리오 후보 대량 생성 → 필터링
```

대표 기법:
- **HAZOP (Hazard and Operability Study)**: 시스템 각 노드에 가이드워드를 적용하여 설계 의도에서 벗어나는 이탈을 체계적으로 식별
- **ODD 분해**: 운행 설계 영역의 파라미터를 축으로 놓고 조합 공간 탐색
- **ISO 34502 시나리오 프레임워크**: Functional → Logical → Concrete 3단계로 구체화

#### HAZOP 가이드워드

HAZOP은 원래 화학/공정 산업에서 시작된 방법론으로, 설계 의도(Design Intent)에서 벗어나는 이탈(Deviation)을 가이드워드로 찾아내는 것이 핵심이다. SOTIF Clause 7의 분석 방법 중 하나로 활용된다.

| 가이드워드            | 의미        | 자율주행 적용 예시              |
| ---------------- | --------- | ----------------------- |
| **No**           | 기능이 아예 없음 | 카메라가 객체를 전혀 탐지하지 못함     |
| **More**         | 과도함       | 레이더가 실제보다 더 많은 객체를 보고   |
| **Less**         | 부족함       | LiDAR 포인트클라우드 밀도가 정상 이하 |
| **Reverse**      | 반대        | 접근 중인 차량을 멀어지는 것으로 판단   |
| **Part of**      | 부분적 수행    | 차선의 일부만 인식              |
| **Other than**   | 의도와 다른 결과 | 보행자를 도로 표지판으로 분류        |
| **Early / Late** | 타이밍 이탈    | AEB 제동 명령이 200ms 지연     |

절차: 시스템을 노드(센서, 인지, 판단, 제어)로 분해 → 각 노드에 가이드워드 적용 → 이탈 시나리오 도출 → 원인·결과·기존 안전장치 기록 → 위험도 판정.

### Data-Driven (경험적): 실제 데이터에서 추출

- **사고 DB**: NHTSA, GIDAS, KOTI 교통사고 데이터에서 시나리오 패턴 추출
- **자연주행 데이터(NDD)**: 대규모 실주행 데이터에서 near-miss, edge case 탐색
- **시뮬레이션 탐색**: 파라미터 공간을 랜덤/적응적으로 샘플링하며 위험 시나리오를 자동 발견 (adversarial search)

## Bottom-Up 시나리오 필터링

Bottom-Up으로 파라미터를 조합하면 수십만 개 후보가 나온다. 전부 시뮬레이션하는 것은 비현실적이므로 두 단계로 필터링한다.

**1단계 — 무의미한 조합 제거.** 물리적으로 불가능하거나 논리적으로 모순인 조합을 규칙 기반으로 자동 제거한다. 예: 눈 + 기온 35°C(물리적 불가능), 고속도로 + 신호 교차로(도로 유형 모순), ODD 정의상 성립하지 않는 조합.

**2단계 — 위험 우선순위 필터링.** 남은 조합 중 어떤 것을 먼저 검증할지 결정한다. Top-Down에서 도출한 인과 체인이 핵심 근거가 된다. 예를 들어 Top-Down이 "폭우 → 카메라 포화 → 보행자 미탐지 → 충돌" 체인을 도출했으면, Bottom-Up 조합 중 "폭우"가 포함된 시나리오에 높은 우선순위를 부여한다.

| 필터링 방법        | 설명                                         |
| ------------- | ------------------------------------------ |
| **인과 체인 매칭**  | Top-Down에서 나온 트리거 조건을 포함하는 조합을 우선 선택       |
| **등가 클래스 축소** | "비/눈/안개"를 "시야 제한" 클래스로 묶어 대표 케이스만 검증       |
| **경계값 집중**    | 파라미터 범위의 극단값(감지 거리 끝자락, ODD 경계) 조합을 우선 |
| **사고 데이터 가중** | 실제 사고 DB에서 빈번한 조건 조합에 높은 가중치               |

Top-Down과 Bottom-Up이 결합되는 지점이 바로 이 필터링이다. 인과 체인 없이 Bottom-Up만 하면 조합을 전부 돌려야 하고, Top-Down만 하면 해저드 목록에 없는 조합을 놓친다.

## 세 접근법의 결합

실제 프로세스에서는 세 접근법이 서로 다른 역할을 수행하며 결합된다.

```
Top-Down (STPA/FTA)      →  "놓치면 안 되는" 시나리오 뼈대 확보
Bottom-Up (HAZOP/ODD 조합) →  파라미터 공간의 커버리지 확보
Data-Driven (사고DB/NDD)   →  현실에서 실제로 일어나는 패턴 반영
         ↓
   시나리오 후보 풀 (수천~수만 개)
         ↓
   위험도 기반 우선순위 필터링
         ↓
   시뮬레이션 / 실차 검증 대상 선정
```

| 접근법             | 강점                      | 한계                  |
| --------------- | ----------------------- | ------------------- |
| **Top-Down**    | 위험한 시나리오만 효율적으로 좁힘      | 해저드 목록에 없는 것은 못 찾음  |
| **Bottom-Up**   | 파라미터 공간 커버리지 체계적        | 조합 폭발, 대부분 무해한 시나리오 |
| **Data-Driven** | 현실 패턴 반영, Unknown 발견 가능 | 데이터에 없는 시나리오는 못 찾음  |

## 4분면과의 관계

시나리오 생성은 본질적으로 Unknown(Area 3)을 Known으로 전환하는 작업이다.

- **Top-Down/Bottom-Up으로 식별된 시나리오** → 즉시 Known으로 이동. 위험하면 Area 2, 안전하면 Area 1.
- **Area 3 (Unknown Hazardous)** → 사전에 목록화할 수 없다. 식별하는 순간 Area 2로 이동한다. 실제 사고, 대규모 시뮬레이션, near-miss에서 사후적으로만 발견된다.
- **SOTIF의 목표** → Area 3의 잔여 위험이 수용 가능한 수준임을 논증하는 것. 직접 열거가 불가능하므로 통계적 논증(충분한 주행 거리/시뮬레이션 커버리지)과 필드 모니터링(Clause 13)으로 간접 입증한다.

## ISO 34502 시나리오 구체화 단계

시나리오는 추상에서 구체로 3단계를 거친다.

| 단계 | 설명 | 예시 |
|---|---|---|
| **Functional** | 자연어 수준의 시나리오 개요 | "선행 차량이 급정거하는 상황" |
| **Logical** | 파라미터 범위가 정의된 시나리오 | "선행 차량 감속도: 4~8 m/s², 차간 거리: 10~30m, 자차 속도: 60~100 km/h" |
| **Concrete** | 모든 파라미터가 특정 값으로 확정 | "감속도: 6 m/s², 차간 거리: 20m, 자차 속도: 80 km/h, 노면: 젖음" |

Bottom-Up 조합은 주로 Logical → Concrete 단계에서 파라미터 공간을 샘플링하는 데 사용되고, Top-Down은 Functional 시나리오를 도출하는 데 사용된다.
