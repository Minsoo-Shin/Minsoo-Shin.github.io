# SOTIF AEB 사례 분석

AEB(자동 긴급 제동)의 고스트 브레이킹(없는 장애물에 급제동)을 예시로 SOTIF 전체 프로세스(Clause 5→13)를 한 바퀴 돌려보는 사례 분석. 실제 제품이 아닌 학습용 시나리오이며, 표준의 각 절이 실무에서 어떤 활동으로 이어지는지를 보여준다.

## Clause 5 — 사양 정의

OEM이 AEB 기능을 정의한다. 표준의 Clause 5.2는 사양에 담을 수 있는 고려 사항을 16개 이상 나열하지만, 이는 "해당되는 경우(where applicable)" 고려하는 항목이며 전부 필수는 아니다. 아래는 학습 목적으로 핵심 6개만 발췌한 것이다 — 실무에서는 자동화 수준, VLSS, 유즈케이스 전환, HMI, 오용 시나리오, 경고·퇴화 전략, 운전자 모니터링, 데이터 수집 절차 등을 추가로 정의한다. 전체 목록은 SOTIF Clause 5 참조.

```
기능: AEB (Automatic Emergency Braking)
ODD: 고속도로, 60~130km/h, 맑음~보통 비
목표: 전방 장애물 감지 시 자동 제동으로 충돌 회피/경감
센서: 전방 카메라 + 전방 레이더
MRC 전략: AEB 오작동 감지 시 기능 비활성화 + 운전자 경고
Driving Policy: TTC(Time To Collision) < 임계값이면 제동 개시
```

## Clause 6 — 해저드 식별 + 위험 평가

"AEB가 할 수 있는 위험한 행동이 뭐가 있는가?"를 체계적으로 식별한다.

| 위험 행동                             | S 평가           | C 평가                    | 판정               |
| --------------------------------- | -------------- | ----------------------- | ---------------- |
| 실제 장애물을 못 보고 미제동 (false negative) | S>0 (충돌)       | C>0 (고속에서 운전자 반응 시간 부족) | **SOTIF 해저드 #1** |
| **없는 장애물에 급제동 (false positive)**  | S>0 (후방 추돌)    | C>0 (뒤차가 못 피함)          | **SOTIF 해저드 #2** |
| 제동력이 너무 약음                        | S>0 (충돌 경감 실패) | C>0                     | **SOTIF 해저드 #3** |

이하 **해저드 #2(고스트 브레이킹)** 를 추적한다. Acceptance criteria를 설정한다:

```
수용 기준: 고스트 브레이킹 발생률 < 1회 / 100만 km
근거: GAMAB — 기존 AEB 미장착 차량 대비 후방 추돌 위험이 증가하지 않아야 함
```

## Clause 7 — 원인 추적

해저드 #2가 "왜 발생하는가"를 양방향으로 파헤친다.

### 시스템 → 시나리오 방향

카메라 분류기가 높은 명암비에서 오분류율이 증가하는 성능 불충분이 있다. 이를 활성화하는 트리거 조건은 "오후 역광".

### 시나리오 → 시스템 방향

고속도로에서 맨홀 뚜껑, 노면 이음새의 그림자, 고가도로 그림자 등 시각적으로 까다로운 조건이 존재한다. 카메라는 이를 장애물로 오인하지만, 레이더는 그림자에 반응하지 않는다 — 센서 융합으로 해결 가능한가?

### Sense-Plan-Act 분석

| 요소        | 기능 불충분                                | 트리거 조건                        |
| --------- | ------------------------------------- | ----------------------------- |
| **Sense** | 카메라 분류기가 노면 그림자를 객체로 오분류              | 오후 역광, 젖은 노면 반사, 고가도로 아래      |
| **Sense** | 레이더가 금속 맨홀 뚜껑에 반사 신호 생성               | 젖은 맨홀, 특정 입사각                 |
| **Plan**  | 센서 융합 알고리즘이 카메라 단독 detection으로도 제동 결정 | 레이더 미감지 + 카메라 high confidence |
| **Act**   | (이 사례에서는 해당 없음)                       | —                             |

### 조합의 중요성

단독 트리거로는 안전하지만 조합이 위험을 만든다:

```
단독: "역광" → 카메라 confidence 저하 → 제동 안 함 (안전)
단독: "젖은 노면" → 약간의 반사 → 제동 안 함 (안전)
조합: "역광 + 젖은 노면" → 카메라가 그림자+반사를 객체로 확신
      → 레이더는 미감지지만 카메라 confidence가 높아서 제동 결정
      → 고스트 브레이킹 발생
```

### 수용 가능성 평가 (7.4) — 인과 체인 추적

각 트리거 조건에 대해 **인과 체인**(트리거 조건 → 기능 불충분 활성화 → 출력 불충분 → 차량 수준 위험 행동)을 끝까지 추적하여 Hazardous/Not Hazardous를 판정한다.

```
트리거 조건
  → 기능 불충분 활성화되는가?
     → No → Not Hazardous (Area 1)
     → Yes → 출력 불충분 발생
        → 차량 수준 위험 행동으로 이어지는가?
           → No → Not Hazardous (Area 1) — 중간에 다른 요소가 차단
           → Yes → Hazardous (Area 2)
```

| 트리거 조건       | 인과 체인                                                  | 판정                |
| ------------ | ------------------------------------------------------ | ----------------- |
| 맑은 날 + 일반 노면 | 카메라 정상 → 기능 불충분 비활성                                    | **Not Hazardous** |
| 역광만 (단독)     | 카메라 오분류 활성화 → 그러나 confidence 0.3 < 임계값 0.7 → Plan에서 차단 | **Not Hazardous** |
| 역광 + 젖은 노면   | 카메라 confidence 0.85 > 임계값 → Plan 통과 → 급제동 → 고스트 브레이킹   | **Hazardous**     |

이 판정은 잠정 추정(estimation)이며, Clause 10에서 실제 테스트로 최종 확인된다. 고스트 브레이킹 시나리오는 **수용 불가** → Clause 8로 진행.

## Clause 8 — 설계 수정

### 시스템 수정 (채택)

**수정 A — 센서 융합 로직 변경.** 카메라 단독 detection으로는 급제동 불가. 레이더가 함께 확인해야 급제동을 허용한다. 카메라만 보이고 레이더에 안 잡히면 경고만 하고 급제동하지 않는다.

**수정 B — 시간적 일관성 검증 추가.** 단일 프레임 detection으로 제동하지 않는다. 최소 3프레임(~100ms) 연속 감지 시에만 제동한다. 그림자는 차량 이동에 따라 빠르게 사라지므로 필터링된다.

### 기능 제한 (부분 채택)

**수정 C — 역광 조건 감지 시 AEB 감도 하향.** 카메라 노출 분석으로 역광을 판단하고, 역광 시 제동 임계값(TTC)을 단축하여 더 확실할 때만 제동한다.

### 사양 업데이트 (Clause 5로 복귀)

```
[업데이트] 제동 판단 조건:
  - 카메라 + 레이더 모두 객체 감지 시 → 즉시 제동
  - 카메라만 감지 시 → 3프레임 연속 + 역광 아닌 조건에서만 제동
  - 역광 감지 시 → TTC 임계값 0.5초 단축
```

## Clause 9 — V&V 전략 수립

수정된 시스템을 어떻게 검증할 것인가 전략을 세운다.

```
Validation target:
  - 고스트 브레이킹 < 1회 / 100만 km (acceptance criteria에서 유도)
  - false negative 증가 없음 (수정으로 인한 회귀 방지)

검증 방법 배분:
  - SIL 시뮬레이션: 500만 km 상당 시나리오 (역광, 그림자, 맨홀 변수 조합)
  - HIL 테스트: 1,000개 트리거 조건 시나리오 (Table 7-C 입력 주입)
  - 테스트 트랙: 50개 재현 가능 시나리오 (실제 역광 + 물 뿌린 노면)
  - 실도로 주행: 10만 km (다양한 기상/시간대)

근거: 시뮬레이션만으로는 카메라의 실제 광학 특성을 완전히 모델링 불가
      → 테스트 트랙과 실도로 비중을 높게 설정
```

## Clause 10 — Known 시나리오 검증

Clause 7에서 찾은 시나리오를 실제로 테스트한다. "아는 위험이 설계 수정으로 실제로 해결됐는가"를 확인하는 단계.

### Sensing 검증

"오후 3시 역광 + 젖은 노면" 조건 재현: 수정 전에는 카메라 confidence 0.92로 제동이 발생했으나, 수정 후에는 레이더 미감지로 제동하지 않음. "고가도로 그림자 + 맨홀 뚜껑" 조건: 3프레임 일관성 체크에서 탈락하여 제동하지 않음(맨홀 레이더 반사는 차량 이동 시 1프레임만 지속).

### Planning 검증 (회귀 테스트)

**실제 장애물은 여전히 잡는가?** 전방 정지 차량 60~130km/h 접근 시나리오에서 카메라+레이더 모두 감지하여 즉시 제동하며, 제동 거리는 수정 전과 동일(false negative 증가 없음).

**카메라만 감지되는 실제 장애물**(블랙 차량, 레이더 반사 약함) 시나리오에서 3프레임 연속 감지 확인 후 제동. 0.1초 지연이 발생하여 130km/h 기준 + 3.6m 제동 거리 증가(130km/h = 36.1m/s × 0.1s)가 있으나, 기존 안전 마진 내로 수용 가능.

### 잔여 위험 평가

SIL 500만 km 시뮬레이션에서 고스트 브레이킹 2건 발견(0.4회/100만 km < 목표 1회/100만 km). 발견된 2건:
- **Case 1**: 대형 트럭의 금속 적재물 + 역광 → 카메라+레이더 모두 오탐
- **Case 2**: 수막 현상으로 노면 전체가 거울 → 3프레임 통과

두 건 모두 새로운 Known 시나리오로 등록하고 설계 루프를 재반복한다.

### 2차 반복

Clause 8(2차)에서 **광학 흐름(optical flow) 분석을 추가** — 실제 객체는 프레임 간 위치가 물리적으로 일관되지만, 반사/환영은 차량 이동에 따라 비물리적 이동 패턴을 보인다. 물리적 일관성 검증 단계를 추가하여 Clause 10(2차)에서 Case 1, 2 재검증 통과.

## Clause 11 — Unknown 시나리오 탐색

"아직 모르는 고스트 브레이킹 원인이 있는가?"

### 탐색 결과

**방법 F (장기 차량 테스트, 10만 km 실 도로)** — 7.2만 km 시점에서 새로운 고스트 브레이킹 1건 발견. 원인: 터널 출구 급격한 조도 변화 + 앞차 브레이크등 잔상. Area 3에서 Area 2로 이동하여 Clause 8에서 수정.

**방법 K (시뮬레이션 무작위 시나리오, 1,000만 km)** — 기상(5종) × 시간대(24h) × 도로유형(4종) × 교통밀도(3단계) 조합. "비+터널 내부 조명+금속 가드레일" 조합에서 1건 추가 발견. Area 3 → Area 2 이동.

**방법 G (플릿 테스트, 차량 50대 × 2만 km = 100만 km)** — 추가 발견 0건.

**방법 I (코너/엣지 케이스)** — "센서 범위 끝자락(150m) + 소형 오토바이 + 역광"에서 false negative 위험 발견. 이건 해저드 #1(미제동) 쪽으로 별도 분석.

총 Unknown 발견 2건 / 1,110만 km. 둘 다 설계 루프로 돌아가 수정 후 재검증 완료. 재검증 후 추가 발견 0건.

| 항목               | 결과                         |

## Clause 13 — 운영 중 모니터링

출시 후에도 계속 감시한다.

모니터링 대상: AEB 작동 로그(제동 시 센서 스냅샷 자동 저장), 운전자가 AEB 직후 즉시 가속한 이벤트(고스트 의심), MRC 전환 발생 횟수, 고객 불만 리포트.

**6개월 후 필드 데이터:** 출하 1만 대 × 평균 1.5만 km = 1.5억 km에서 고스트 브레이킹 12건 보고(0.08회/100만km). 수용 기준 충족. 그러나 **새로운 패턴 발견** — "3D 착시 횡단보도(입체 페인팅)"가 설치된 도로에서 3건 집중 발생. 출시 전에는 이 도로가 없었다(컨텍스트 진화). 설계 루프로 복귀하여 OTA 업데이트로 수정을 배포한다.

## 4분면 변화 추적

각 Clause를 거치면서 4분면이 어떻게 변하는지 추적한다. 해저드 #2(고스트 브레이킹)에 한정하여 시나리오 수로 표현한다.

**주의**: 4분면은 표준이 "conceptual approach"이며 "sizes are not measurable"라고 명시한 개념적 프레임워크다. 특히 Unknown 영역(Area 3, 4)은 **발견 전에는 서로 구분할 수 없다** — 시나리오를 발견하고 인과 체인을 추적한 결과 Hazardous면 "Area 3에 있었던 것", Not Hazardous면 "Area 4에 있었던 것"으로 **사후적으로** 분류되는 것이다. 아래 표의 "Area 3 → Area 2", "Area 4 → Area 1" 이동은 이 사후적 관점에서 서술한다.

### Clause 5 후 — 거의 모든 것이 Unknown

사양만 정의한 상태. 어떤 시나리오가 안전하고 위험한지 아직 모른다.

```
          Known          Unknown
     ┌──────────────┬──────────────┐
 Safe│  Area 1: 0   │  Area 4: ?   │
     ├──────────────┼──────────────┤
 Haz │  Area 2: 0   │  Area 3: ?   │
     └──────────────┴──────────────┘
```

Area 3과 4에 시나리오가 존재하지만, 아직 아무것도 식별하지 않았으므로 전부 Unknown이다.

### Clause 6 후 — 위험 "행동"은 알지만, 구체적 시나리오는 아직 모름

차량 수준에서 "고스트 브레이킹이 위험하다"(S>0, C>0)는 것을 확인했다. 하지만 **"어떤 조건에서 발생하는가"는 아직 모르므로, 4분면의 구체적 시나리오 이동은 거의 없다.**

```
          Known          Unknown
     ┌──────────────┬──────────────┐
 Safe│  Area 1: 0   │  Area 4: ?   │
     ├──────────────┼──────────────┤
 Haz │  Area 2: ~0  │  Area 3: ?   │  ← "고스트 브레이킹이 위험하다"는 알지만
     └──────────────┴──────────────┘     구체적 트리거 조건은 Clause 7에서 식별
```

S=0이나 C=0으로 판정된 위험 행동(예: 저속 주차 보조 10cm 편차)은 이 시점에 4분면 바깥으로 제외된다 — 추가 분석 불필요.

### Clause 7 후 — 시나리오가 구체화되어 Area 3 → Area 2 대량 이동

기능 불충분 + 트리거 조건을 양방향 분석하여 구체적 시나리오가 드러난다.

```
          Known          Unknown
     ┌──────────────┬──────────────┐
 Safe│  Area 1: 3   │  Area 4: ?   │  ← 분석 결과 안전한 것들
     ├──────────────┼──────────────┤     (역광만, 젖은 노면만 등 단독 트리거)
 Haz │  Area 2: 4   │  Area 3: ?   │  ← 위험 확인된 조합들
     └──────────────┴──────────────┘
```

| 이동                  | 시나리오                             | 이유                |
| ------------------- | -------------------------------- | ----------------- |
| Area 3 → **Area 2** | 역광 + 젖은 노면 → 고스트 제동              | 위험 조합 식별          |
| Area 3 → **Area 2** | 고가도로 그림자 + 맨홀 반사                 | 위험 조합 식별          |
| Area 3 → **Area 2** | 카메라 단독 high confidence + 레이더 미감지 | Plan 불충분 식별       |
| Area 3 → **Area 2** | 젖은 맨홀 + 특정 입사각                   | 레이더 오탐 식별         |
| Area 4 → **Area 1** | 역광만 (단독)                         | 분석 결과 제동 안 함 = 안전 |
| Area 4 → **Area 1** | 젖은 노면만 (단독)                      | 분석 결과 제동 안 함 = 안전 |
| Area 4 → **Area 1** | 맑은 날 + 일반 노면                     | 분석 결과 오탐 없음 = 안전  |

핵심: Clause 7은 Area 3의 시나리오를 **Area 2로 끌어내는** 단계다. Area 3 자체가 줄어든다.

### Clause 8 후 — Area 2의 시나리오를 설계로 해결 (검증 대기)

설계 수정(센서 융합 + 시간 일관성 + 역광 감도)으로 Area 2의 시나리오들을 Area 1로 보내려는 **의도**는 있지만, 실제로 해결됐는지는 아직 검증하지 않았다. 4분면에서 이동은 **보류 상태**.

```
          Known          Unknown
     ┌──────────────┬──────────────┐
 Safe│  Area 1: 3   │  Area 4: ?   │
     ├──────────────┼──────────────┤
 Haz │  Area 2: 4   │  Area 3: ?   │  ← 설계는 수정했지만 검증은 아직
     └──────────────┴──────────────┘     (Area 2 → Area 1 이동 보류)
```

### Clause 9 — 4분면 변화 없음

전략 수립만 하는 단계. "Area 2를 어떻게 검증할 것인가, Area 3을 어떻게 탐색할 것인가"를 계획한다.

### Clause 10 후 (1차) — Area 2 → Area 1 확정, 그러나 새로운 Area 2 추가 발견

검증을 통과한 시나리오는 Area 2에서 Area 1로 **확정** 이동한다.

```
          Known          Unknown
     ┌──────────────┬──────────────┐
 Safe│  Area 1: 7   │  Area 4: ?   │  ← 기존 4건 중 4건 검증 통과 (+4)
     ├──────────────┼──────────────┤
 Haz │  Area 2: 2   │  Area 3: ?   │  ← 하지만 검증 중 2건 새로 발견
     └──────────────┴──────────────┘     (금속 적재물, 수막 현상)
```

| 이동                  | 시나리오                 | 이유                       |
| ------------------- | -------------------- | ------------------------ |
| Area 2 → **Area 1** | 역광 + 젖은 노면           | 센서 융합 수정으로 제동 안 함, 검증 통과 |
| Area 2 → **Area 1** | 고가도로 그림자 + 맨홀        | 3프레임 일관성으로 필터링, 검증 통과    |
| Area 2 → **Area 1** | 카메라 단독 detection     | 융합 로직 수정으로 급제동 차단, 검증 통과 |
| Area 2 → **Area 1** | 젖은 맨홀 반사             | 시간 일관성으로 필터링, 검증 통과      |
| Area 3 → **Area 2** | 금속 적재물 + 역광 (Case 1) | SIL 시뮬레이션에서 **새로 발견**    |
| Area 3 → **Area 2** | 수막 현상 노면 거울 (Case 2) | SIL 시뮬레이션에서 **새로 발견**    |

검증 과정에서 시뮬레이션이 Area 3을 추가로 탐색한 셈이다. 새로 발견된 2건을 해결하기 위해 **설계 루프 2차 반복**(optical flow 추가) 후 재검증.

### Clause 10 후 (2차) — 추가 발견분도 해결

```
          Known          Unknown
     ┌──────────────┬──────────────┐
 Safe│  Area 1: 9   │  Area 4: ?   │  ← Case 1, 2 도 해결 (+2)
     ├──────────────┼──────────────┤
 Haz │  Area 2: 0   │  Area 3: ?   │  ← Known Hazardous 전부 해소
     └──────────────┴──────────────┘
```

이 시점에서 **알고 있는 위험 시나리오는 전부 해결**됐다. 하지만 Area 3(모르는 위험)은 여전히 존재할 수 있다.

### Clause 11 후 — Area 3 축소

Unknown 시나리오 탐색으로 Area 3을 줄인다.

```
          Known             Unknown
     ┌───────────────┬───────────────┐
 Safe│  Area 1: 11   │  Area 4: ?    │  ← 탐색 중 안전 확인된 것들도 Area 1로
     ├───────────────┼───────────────┤
 Haz │  Area 2: 0    │  Area 3: ↓↓↓  │  ← 1,110만 km 탐색으로 대폭 축소
     └───────────────┴───────────────┘
```

| 이동                           | 시나리오         | 이유              |
| ---------------------------- | ------------ | --------------- |
| Area 3 → Area 2 → **Area 1** | 터널 출구 조도 변화  | 발견 → 수정 → 검증 통과 |
| Area 3 → Area 2 → **Area 1** | 비+터널 조명+가드레일 | 발견 → 수정 → 검증 통과 |

Area 3이 **정확히 0이 될 수는 없다** — 현실의 모든 시나리오를 탐색하는 것은 불가능하기 때문이다. 그래서 Clause 11의 목표는 "Area 3 = 0"이 아니라 **"Area 3의 잔여 위험이 수용 기준 이하임을 충분한 신뢰도로 논증"**하는 것이다. 1,110만 km에서 2건만 발견되었고 모두 해결됐다는 것이 그 논증의 근거가 된다.

### Clause 12 — 최종 상태 논증

```
          Known             Unknown
     ┌───────────────┬───────────────┐
 Safe│  Area 1: 11+  │  Area 4: ?    │  ← 최대화됨
     ├───────────────┼───────────────┤
 Haz │  Area 2: 0    │  Area 3: ≈ε   │  ← 잔여 위험 < 수용 기준
     └───────────────┴───────────────┘

릴리스 논증:
  Area 2 = 0 (모든 Known Hazardous 해결 완료)
  Area 3 ≈ ε (1,110만 km 탐색, 잔여 위험 < 1회/100만 km)
  → "Area 2 + Area 3의 잔여 위험이 수용 기준 이하" → 승인
```

### Clause 13 — 현실이 4분면을 다시 흔든다

출시 후 6개월, 3D 착시 횡단보도가 새로 설치된 도로에서 고스트 브레이킹 발생.

```
          Known             Unknown
     ┌───────────────┬───────────────┐
 Safe│  Area 1: 11+  │  Area 4: ?    │
     ├───────────────┼───────────────┤
 Haz │  Area 2: 1    │  Area 3: ?    │  ← 3D 횡단보도: 신규 Area 2
     └───────────────┴───────────────┘
       컨텍스트 진화로 새 시나리오 출현
       → 설계 루프 복귀 → OTA 수정
```

이 시나리오는 출시 전에 **존재하지 않았던** 조건이다(도로 인프라 변화). Area 3에 있다가 발견된 것이 아니라, **세계 자체가 변해서 새로운 시나리오가 생긴 것**. 표준이 Clause 13에서 "컨텍스트 진화(도로·교통 변화, 규제 변경, 새로운 유형의 오용)"를 모니터링하라고 요구하는 이유다.

## 사이클 요약

```
설계 루프 1차: Clause 5→6→7→8 (센서 융합 + 시간 일관성 + 역광 감도)
  ↓
V&V 루프 1차: Clause 9→10 (Known 검증에서 2건 추가 발견)
  ↓
설계 루프 2차: Clause 8 (optical flow 추가)
  ↓
V&V 루프 2차: Clause 10→11 (Unknown 탐색에서 2건 추가 발견)
  ↓
설계 루프 3차: Clause 8 (터널 조도 + 가드레일 반사 대응)
  ↓
V&V 루프 3차: Clause 10→11 (재검증, 추가 발견 0건)
  ↓
Clause 12: 릴리스 승인
  ↓
Clause 13: 운영 모니터링 → 6개월 후 3D 횡단보도 이슈 → 설계 루프 복귀
```

한 번에 끝나지 않는다. 이 사례에서는 Clause 10에서 2건, Clause 11에서 2건, Clause 13에서 1건을 추가 발견하여 총 3번 설계 루프를 돌았다. **반복이 SOTIF의 본질**이다.
